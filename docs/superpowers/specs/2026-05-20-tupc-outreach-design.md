# TUPC Cold Outreach Tool — Design Spec

**Date:** 2026-05-20
**Status:** Approved for implementation
**Owner:** Lucas Reydman (technical), handed off to Zach (TUPC marketing intern) for daily use

---

## Problem

Toronto United Pickleball Club (TUPC, a CNPL pro pickleball team) needs to run sponsorship outreach to aligned Canadian consumer brands. Today the work is fully manual: each email written from scratch in Gmail, the 14-page sponsorship deck attached by hand, no follow-up cadence, no record of who replied. Zach — a marketing intern, not a developer — is the operator.

The original PRD (ChatGPT-drafted) described an enterprise-grade system: standalone CRM, brand intelligence layer, full automation engine, KPI dashboard. That's massively overscoped for one intern at one team sending 20–50 emails a week. This spec defines the minimum viable system that actually achieves the goal: get Zach sending personalized, deck-attached cold emails with automated follow-ups and reply tracking, in a tool Lucas can hand off in under five minutes.

## Goals

- Zach manages a list of target brands in one place and sends a personalized cold email per row with one click.
- Every email is **freshly generated by Claude** with brand-specific positioning — no merge-field templates that read like mail-merge.
- Follow-up sequence runs automatically; inbound replies stop the sequence and flip status.
- Lucas hands the tool to Zach in <5 minutes of setup; Zach can use it without any developer help.
- Tool is open source on GitHub.

## Non-goals (v1)

- Open-rate pixel tracking — unreliable on Apple Mail and Gmail's image proxy.
- Multi-user, role-based access — single operator.
- Standalone web app — Sheet is the UI.
- Pre-built brand intelligence database (priorities, past sponsorships) — Claude personalizes on demand, no database to maintain.
- Proposal builder, ROI reports, calendar integration — explicitly future.
- Cold-email infrastructure (warmup, domain rotation, SMTP pools) — Gmail handles this volume.
- A/B testing of templates — premature.

## Constraints

- Hand-off-able to a non-developer. Setup wizard for first run; everything else lives in the Sheet.
- Volume: 20–50 emails/week, well under Gmail's 500/day free or 2,000/day Workspace limit.
- CASL (Canada's anti-spam law) compliance: every email must include sender identification, business address, and a working unsubscribe.
- Uses the existing TUPC 2026 Deck PDF as-is.
- Anthropic API costs paid via Lucas's account; key pre-filled in Sheet Config before hand-off.

**Why this and not a web app:** zero hosting, zero deployment, zero install. Lucas shares one Sheet with Zach — done. Apps Script natively handles Gmail (send, attach Drive files, label, scan threads) and time-based triggers. The Sheet IS the CRM; an intern already knows spreadsheets. Source-controllable via `clasp` so the GitHub-repo requirement is satisfied. A web app would mean hosting, a domain, Gmail OAuth wiring, and a UI build — all unnecessary for the actual use case.

### Apps Script modules

- **`Main.gs`** — `onOpen()` installs custom menu; menu handlers; sidebar entry point for draft review.
- **`SetupWizard.gs`** — first-run flow that prompts for API key, deck file, sender info; writes to Config; installs time triggers.
- **`Outreach.gs`** — `generateDraft(rowIndex)`, `sendDraft(rowIndex)`, `processFollowUps()`. Owns email composition + status transitions + send-cap enforcement.
- **`ClaudeClient.gs`** — `callClaude(systemPrompt, userMessage)` wrapper around `UrlFetchApp` to `api.anthropic.com/v1/messages`. Auth, retries, structured error surface.
- **`GmailScanner.gs`** — `scanForReplies()` iterates Brands rows with `thread_id`, fetches thread, marks `replied` if inbound message exists after the latest outbound.
- **`SheetSetup.gs`** — `installSheetStructure()` idempotent: creates tabs, headers, dropdowns, data validation, frozen rows, default Templates rows, default Config rows.
- **`Triggers.gs`** — `installTimeTriggers()` idempotent: daily 9am `processFollowUps()`, hourly `scanForReplies()`.
- **`Sidebar.html`** — draft preview/edit UI shown after generation.

### Email composition (the core IP)

For each send, Claude receives:

**System prompt** — embedded as `templates/tupc-brand-context.md`. Contents:
- TUPC positioning: CNPL champion 2023 / finalist 2025, 10M+ impressions S3, 1M+ combined social, partnerships from $3,500, partners (Roots, Sleeman, Club Med, Cadillac Fairview, TSS).
- Three case studies with concrete results (Club Med 80k views / 10 episodes; Cadillac Fairview 5k in-person + 100k digital; Roots 100k digital).
- Pickleball stats: +291% Canadian participation since 2021, 1.54M Canadian players, +57% growth since 2022.
- Writing rules: ≤120 words; plain language; brand-specific opener (not "I hope this finds you well"); one specific activation idea tied to the brand's category; soft CTA for a 15-minute call; no marketing fluff.

**User message** (per send, assembled by `Outreach.gs`):
- Brand: `{company}` (`{category}`)
- Recipient: `{contact_name}`, `{contact_role}`
- Pitch angle (if provided): `{pitch_angle}`
- Stage: `{initial | follow_up_1 | follow_up_2 | breakup}`
- Sender: `{sender_name}`, `{sender_title}`

**Output format**: Claude returns strict JSON `{"subject": "...", "body": "..."}` so parsing is mechanical. Validate `subject.length < 60` and `body` contains `sender_name`; on validation failure, retry once, then mark `draft_failed`.

The signature, CASL business address, and unsubscribe mailto are appended by `Outreach.gs` after Claude's body — they are non-negotiable and not delegated to the LLM.

### Send flow

1. Zach selects a row, clicks **Generate draft** in the menu.
2. `generateDraft` calls Claude, writes `draft_subject` + `draft_body` to the row, sets `status = drafted`, opens the sidebar.
3. Zach edits if desired; clicks **Send** in the sidebar.
4. `sendDraft`:
   - Composes a MIME message with the deck PDF attached from Drive (`DriveApp.getFileById(config.deck_drive_file_id).getBlob()`).
   - Appends signature, business address, unsubscribe.
   - Sends via `GmailApp.sendEmail(to, subject, plain, {htmlBody, attachments, cc, name})`.
   - Captures the thread via `GmailApp.search('from:me to:{contact_email}', 0, 1)[0].getId()` (or by reading user's Sent label).
   - Sets `status = sent`, `last_action_date = today`, `next_action_date = today + day_offset_for_follow_up_1`, `sent_count += 1`.
   - Clears `draft_subject` / `draft_body` (full history visible in Gmail Sent).

### Follow-up flow

`processFollowUps()` runs daily at 9am Eastern:

1. Count today's sends so far (rows with `last_action_date = today`). If that count ≥ `daily_send_cap`, exit immediately — do not burn Claude API calls on rows that can't send today.
2. Find rows where `next_action_date <= today` AND `status in (sent, follow_up_1, follow_up_2)` AND `sent_count < 4`. Process in `next_action_date` order (oldest first), stopping if the cap is hit mid-loop.
3. For each row, determine the next stage from current status:
   - `sent → follow_up_1`
   - `follow_up_1 → follow_up_2`
   - `follow_up_2 → breakup` (the breakup email itself is a send action; after it lands, the row's status becomes `dead`, not `breakup`. `breakup` is never a persisted status — it's the stage name passed to the Claude prompt.)
4. Generate stage-specific email via Claude and send as a reply on the existing thread (`GmailApp.getThreadById(thread_id).reply(body, {htmlBody, attachments})`).
5. Update status to the new stage (or `dead` after a breakup send) and set `next_action_date = today + next_stage_offset`.

### Reply detection

`scanForReplies()` runs hourly:

1. For each Brands row with non-empty `thread_id` and `status ∉ (replied, meeting_booked, won, dead, invalid_email, send_failed, draft_failed)`:
2. Fetch `GmailApp.getThreadById(thread_id)`.
3. Find the most recent outbound message in the thread (`message.getFrom()` matches the authorized sender). Compare against actual `Message.getDate()` timestamps, not the Sheet's `last_action_date` — Sheet dates are dates, Gmail timestamps are datetimes, and a reply landing the same day as a send would otherwise be missed.
4. If any inbound message (`getFrom()` not matching sender) has a timestamp later than the most recent outbound, set `status = replied`, `reply_at = the inbound message's timestamp`, and apply Gmail label `TUPC/Replied` to the thread.

### Setup wizard (first-run UX)

On `onOpen()`, check if `Config.anthropic_api_key` is set:

- If not, the custom menu shows only **🎾 Run setup**.
- The wizard is an HTML sidebar with screens:
  1. Welcome — what this does
  2. Anthropic API key (paste in)
  3. Gmail authorization (button triggers the permission grant)
  4. Drive deck file (file picker, stores file ID)
  5. Sender info (name, title, CC, signature)
  6. Confirmation + **Send test email to yourself** button
- On completion: installs time triggers, replaces the menu with the full outreach menu.

If Lucas pre-fills the Config tab before hand-off (intended path), Zach only does step 3 (Gmail auth) and step 6 (test send) — the wizard skips ahead.

## CASL compliance

Every outbound email gets appended:

```
—
{sender_signature}

Toronto United Pickleball Club
{business_address}

To stop receiving these emails: {unsubscribe_mailto}
```

`Outreach.gs` appends this on every send (initial and follow-ups). Tests verify presence in the final body.

## Error handling

- **Claude API failure** (network, 4xx, malformed JSON): toast in Sheet UI, set `status = draft_failed`, leave row unchanged otherwise. One retry with backoff before marking failed.
- **Gmail send failure** (quota, recipient bounce caught synchronously): toast, set `status = send_failed`, do not advance counters.
- **Reply scan rate limit**: catch, log, back off; non-fatal.
- **Invalid recipient email** (regex check before any work): set `status = invalid_email`.

## Testing strategy

Apps Script has no native test runner, so we hand-roll one:

- `tests/` contains `.gs` files with functions named `test_*`. These files DO get pushed by `clasp` alongside `src/` — Apps Script projects are flat. To keep Zach from invoking them accidentally:
  - Test functions never appear in the custom menu.
  - `runAllTests()` is only callable from the Apps Script editor (no menu entry, no trigger).
  - All test files are prefixed `Test_` so they sort to the bottom of the editor file list.
- `runAllTests()` entrypoint executes them, writes pass/fail to a `Test Results` tab.
- Test doubles:
  - `FakeClaude` returns canned JSON without calling the API.
  - `FakeGmail` writes intended sends to a `Test Outbox` tab.
- Coverage targets:
  - `SheetSetup.installSheetStructure` is idempotent (re-running doesn't duplicate).
  - `Outreach.generateDraft` parses good JSON, retries on malformed, marks `draft_failed` on second failure.
  - `Outreach.sendDraft` advances status and dates correctly.
  - `Outreach.processFollowUps` selects the right rows, respects `daily_send_cap`, halts on `breakup`.
  - `GmailScanner.scanForReplies` flips status only when an inbound message post-dates the last outbound.
  - CASL footer is present on every composed body.
  - Invalid email triggers `invalid_email` status without API calls or sends.

## Risks

- **Deliverability**: At 20-50/wk on a real Gmail with personalized content, expect 85–95% inbox placement. Mitigation: vary send times, don't blast consecutive emails to the same domain. If volume scales 10×, recommend moving to Smartlead/Instantly — out of v1 scope.
- **API key visibility in Sheet**: Anyone with editor access on the Sheet can read the Anthropic key. Mitigation: only share with Zach. Future: move to `PropertiesService` (per-user script properties) — not v1 because it complicates hand-off (requires Zach to paste the key himself).
- **Claude rate limits**: At ~50/wk this is a non-issue.
- **CASL violations**: Mitigated by always-on footer, verified by test.
- **Gmail thread ID capture**: searching by recipient immediately after send is the documented pattern but has a race window. Mitigation: small `Utilities.sleep(2000)` after send before the search; retry up to 3× if not found. If still not found, set `thread_id = "MISSING"` and skip follow-ups for that row (logged in `notes`).
